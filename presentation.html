How our document Q&A service actually works

A walkthrough for engineers — no RAG background needed

Goal: by the end everyone understands what RAG is, how this repo implements it, and where money (tokens) gets spent. Keep it concrete.

The whole thing in one sentence

Take a PDF → break it into pieces → store them in a searchable database → when someone asks a question, find the relevant pieces and ask an LLM to answer using only those pieces.

🧠 Analogy: It's an open-book exam for the AI. Instead of answering from memory, it must point to the page in our document. No page = no answer.
This is the anchor. Everything else is detail on these two halves: retrieval (find pieces) and generation (write answer).

What is RAG?

Retrieval-Augmented Generation

Retrieval

Search our documents and pull out the few passages most relevant to the question.

Augmented

Inject those passages into the LLM's prompt as context.

Generation

The LLM writes a natural-language answer — grounded in what we gave it.
Plain LLM
Answers from training memory. Doesn't know our PDFs. Can confidently make things up (“hallucinate”). Can't cite a source.
RAG
Answers from our documents. Can cite source + page. Says “not in the docs” instead of guessing.
The key win: grounding + citations + a clean refusal path. That's why enterprises use RAG instead of fine-tuning for document Q&A.

Why not just paste the whole PDF into the LLM?

  • 📏 It may not fit — big docs blow past the context window.
  • 💸 Cost — you'd pay to send the entire document on every question.
  • 🎯 Accuracy — burying the answer in 100 pages of noise makes the model worse, not better.
  • 🔁 Reuse — ingest once, ask thousands of questions cheaply.
RAG's trick: send the LLM only the 4 paragraphs that matter, not the whole book.
This motivates chunking + retrieval. The "send only what matters" line is the cost story in a nutshell.

The RAG protocol — two phases

① INGEST  — done once per document

1Load
read PDF text, keep page #s
→
2Chunk
cut into ~800-char overlapping pieces
→
3Embed
turn each chunk into a vector
→
4Store
save vectors + text in vector DB

all local · free · no tokens

② QUERY  — done every question

1Embed
turn question into a vector
→
2Search
find top-k closest chunks
→
3Generate
LLM answers from those chunks
→
4Cite
attach source + page

local · free   ⚡ step 3 = Gemini = the only paid step

This is THE slide. Green = local/free, amber = paid. Note the symmetry: same embedding model used in both phases. Only generation calls an external API.

What's an embedding? (the one new concept)

A model converts a piece of text into a list of numbers (a vector, ~384 of them) that represents its meaning.

Texts about similar topics get similar vectors — so “close numbers” ≈ “close meaning”.

Search = “which stored chunks are numerically closest to the question?” This is semantic search — matches meaning, not keywords.

Think of a map 🗺️

Every chunk gets a coordinate. “invoice” and “billing” land near each other even though the words differ.

A question becomes a coordinate too — we just grab its nearest neighbours.
💡 In this project, embeddings run on a local model (all-MiniLM-L6-v2) on our own machine — free, private, no API call.
Don't go deeper than "numbers that capture meaning, close = similar". The map analogy lands well with programmers (think k-nearest-neighbour).

How our repo implements it

🚫 Myth-busting

  • No LlamaIndex / LangChain. Hand-built from small libraries → less magic, easy to debug.
  • The LLM is Google Gemini, not Claude. Tokens here = Gemini tokens.

🧱 The pieces

Two endpoints: POST /ingest  ·  POST /query  ·  persists to ./data/chroma

Reassure the team: nothing exotic. If you can read Python you can read this whole repo in 20 minutes. Vector DB lives on disk, survives restarts.

Where each step lives in the code


app/
├─ main.py                 # wires up the two endpoints + /health
├─ config.py               # knobs: chunk_size, top_k, model names, API key
├─ routers/
│  ├─ ingest.py            # PHASE 1: load → chunk → embed → store
│  └─ query.py             # PHASE 2: embed → search → LLM → cite
└─ services/
   ├─ pdf.py               # extract text per page (pypdf)
   ├─ chunking.py          # ~800 chars, 150 overlap, never mid-word
   ├─ embeddings.py        # LOCAL model → vectors           [free]
   ├─ vectorstore.py       # ChromaDB upsert + nearest search [free]
   └─ llm.py               # Gemini prompt + answer        [TOKENS]

Read in this order: ingest.py → query.py → llm.py

Point them at llm.py specifically — it's where the prompt and the grounding rules live, and the only file that talks to a paid API.

The prompt we send Gemini (services/llm.py)


SYSTEM: Answer ONLY from the CONTEXT below. Do NOT use prior knowledge.
        If the answer isn't there, reply EXACTLY:
        "I can't answer that from the provided documents."
        Cite facts inline as [source, page N].

CONTEXT:
[RFE letter, page 3]   <chunk text…>
[RFE letter, page 5]   <chunk text…>
… (the top-4 retrieved chunks) …

QUESTION: <the user's question>
  • temperature = 0 → deterministic, no creative drift.
  • If the model returns that refusal line → we mark grounded:false & drop citations.
The system rules are what make it "grounded RAG" vs. a chatbot. The exact-match refusal sentence is how the code detects "not found".

💸 Where do tokens (cost) get used?

A token ≈ ¾ of a word. APIs bill per token in + per token out.

StepLLM tokens?Cost
PDF read · chunkNofree (local)
Embedding (ingest + query)Nofree — local model
Vector searchNofree (local DB)
Generation — /queryYES — Geminipaid, per question
🔑 Ingesting a 100-page PDF costs ZERO tokens. Only asking questions costs money — and only the final answer step.
This is the slide they'll quote back to management. Hammer: ingest free, query paid. Embeddings being local is the surprise for most people.

One question's bill — and the levers

What you pay for per question

  • Input: system rules + 4 chunks + question
    ≈ 4 × ~800 chars ≈ ~900–1,000 tokens
  • Output: length of the answer
The retrieved chunks dominate the input cost.

Knobs to cut cost

  • ⬇️ top_k (default 4) — fewer chunks = fewer input tokens
  • ⬇️ chunk_size (default 800)
  • ✅ already on the cheap tier: gemini-2.5-flash
  • ♻️ re-ingesting is free — questions are the cost
All in app/config.py / .env
top_k is the single biggest cost/quality dial. More chunks = better recall but more tokens. 4 is a sane default.

Live demo


# 1. start the service
uvicorn app.main:app --reload

# 2. ingest the sample PDF  →  one-time, FREE
curl -X POST localhost:8000/ingest \
  -H 'content-type: application/json' \
  -d '{"path": "Compressed RFE letter (2).pdf"}'
# → { source, pages, chunks_added }

# 3. ask a question  →  spends Gemini tokens
curl -X POST localhost:8000/query \
  -H 'content-type: application/json' \
  -d '{"question": "What does the letter ask for?"}'
# → { answer, grounded: true, citations: [{source, page, snippet}] }

Then ask something not in the PDF → watch it refuse instead of guessing.

If live demo is risky, screenshot the two responses beforehand. Show the citations array — that's the "grounding" payoff made visible.
